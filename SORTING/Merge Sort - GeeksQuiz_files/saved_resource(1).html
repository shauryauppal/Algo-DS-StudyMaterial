                      rcgldr Arnab • 3 months ago

                      On a processor with 16 registers, like a X86 in 64 bit mode, a 4 way bottom up merge sort on an array of integers ends up slightly faster than 2 way. The total number of operations is the same, 1.5 x comparisons, 0.5 x moves, but the comparisons are more cache friendly, so the 4 way bottom up merge sort ends up about 15% faster, about the same as quicksort.

                        rcgldr Archita • 7 months ago

                        In the case of a bottom up merge sort, the list is divided into n (n == length) runs of size 1 in the case of an all merge sort, or in n/32 runs of size 32 in the case of a hybrid insertion sort (sorts 32 element runs) / merge sort (merges the runs), but only pairs of runs are merged at a time, merging pairs of runs from one array to another (temp) array, which doubles the run size (1, 2, 4, 8, ... ), merging from temp array back to original array, repeating the process until run size >= length.

                        Again in the case of a bottom up merge sort, on a machine with 16 registers, like a PC in 64 bit mode, in some cases, a 4 way merge sort with a bunch of working pointers, almost all of them in registers, may be a bit faster than a 2 way merge sort. The number of basic operations are the same: 2 way averages 1 compare and 1 move for each element merged, 4 way averages 1.5 times the number of compares and 0.5 the number of moves (3 compares + 1 move per element merged) / 2 (only 1/2 the number of merge passes). The 4 way is only a bit faster if compares are a bit faster than moves.

                                                                                      rcgldr rahul singh • 3 months ago

                                                                                      sort() recursively divides the array until a sub-array has size 1, in which case a sub-array of size 1 can be considered to be sorted, and the merging process begins. The order of merging is depth first, left first. A bottom up merge sort (not shown) skips the unneeded recursion to split up sub-arrays and starts off by considering an array of size n to be n sub-arrays of size 1, and starts merging immediately, using iteration instead of recursion. Most libraries, such as Java's built in sort for objects, use some variation of bottom merge sort, such as timsort, which splits up an array into k sub-arrays of size n/k, uses insertion sort on each of the k sub-arrays, then uses normal bottom up merge sort to merge the k sub-arrays.
